Vulkan Multi-GPU Split Mode Graph for llama.cpp

NVIDIA has NCCL for multi-GPU LLM inference. AMD consumer GPUs on PCIe have nothing. This project adds multi-device Vulkan support to ik_llama.cpp's graph scheduler, enabling split-mode inference across multiple AMD GPUs using only standard Vulkan 1.2+ APIs.

Vendor-neutral. Works on RADV, NVIDIA proprietary, ANV, lavapipe. Tested on hardware spanning 2010 (Xeon X5650 + Polaris 12) to 2019 (Ryzen 9 + RDNA 2).

12 Phases
~1,200 Lines changed

Problem

CUDA path. Direct GPU-to-GPU transfers via cudaMemcpyPeerAsync over PCIe or NVLink. NCCL handles multi-GPU communication transparently. Split mode graph scheduling works out of the box.
Vulkan path. No native peer-to-peer. The Vulkan backend has infrastructure for multiple devices—each VkDevice maps to one ggml_backend—but the async functions were wired to NULL and no cross-device copy mechanism existed. All the plumbing for multi-GPU was there. None of it was connected.

Phases

Each phase is independently upstreamable. Foundation phases (1–6) benefit single-GPU Vulkan too.

1 Async Interface and Events complete
  • Un-NULL the async vtable entries: set_tensor_async, get_tensor_async, cpy_tensor_async, synchronize
  • Implement event API using Vulkan timeline semaphores
2 Split Buffer Type complete
  • Distribute tensor rows across multiple Vulkan devices, mirroring CUDA's split buffer pattern
  • Handle quant-interleaved row layouts for Q2_K through Q8_0
3 Cross-Device Copies via Host Staging complete
  • GPU A → host staging → GPU B. The only portable approach in the Vulkan spec
  • Staging buffer pool to avoid repeated allocation
4 Async Execution and Pipeline Parallelism complete
  • Non-blocking graph_compute: submit to GPU, return immediately
  • Thread safety audit of per-context state vs shared vk_instance
5 Topology Discovery complete
  • Device type awareness: discrete GPU, integrated, CPU (lavapipe)
  • VRAM reporting and multi-GPU topology logging at startup
6 Runtime Testing and Fallbacks complete
  • Scalar fallbacks for IQK on non-AVX2 hosts (Westmere Xeon)
  • lavapipe fixes: memcpy early-outs for host-visible buffers
7 Async Cross-Device Copy Pipeline complete
  • Replace synchronous fence wait with deferred pipeline: submit read, defer wait to synchronize()
  • 17.8x prompt eval speedup on retro hardware
8 Double-Buffered Staging complete
  • Per-copy staging pool with individual fences (correctness fix for concurrent copies)
9 Parallel Split Buffer Uploads complete
  • Threaded per-device model weight uploads during loading
10 True Async Graph Compute complete
  • graph_compute returns after vkQueueSubmit; fence wait deferred to synchronize()
  • Overlaps CPU graph preparation with GPU execution
11 Transfer Queue Utilization skipped
  • Cross-device copies already use transfer queues (Phase 7)
  • No DMA/compute overlap opportunity with only 3 graph splits
12 dmabuf Zero-Copy Transfer complete
  • Share GPU buffers via Linux dmabuf file descriptors, eliminating CPU memcpy
  • VK_KHR_external_memory_fd + VK_EXT_external_memory_dma_buf

Data Flow

Before (Phases 3–10). Every cross-device copy does 4 data moves—2 DMA transfers and 2 CPU memcpys through host staging:
GPU A device-local → [DMA] → host staging → [CPU memcpy] → host staging → [DMA] → GPU B device-local
After (Phase 12). dmabuf shares physical pages between devices. 2 DMA transfers, zero CPU involvement:
GPU A device-local → [DMA] → shared dmabuf → [DMA] → GPU B device-local

Transfer Overhead

Per-token cross-device transfer cost, measured on RX 6800 XT + Vega:

TinyLlama 1.1B — Phase 10 (host staging) +5 ms
TinyLlama 1.1B — Phase 12 (dmabuf) −0.7 ms
Llama-2-7B — Phase 10 (host staging) +15 ms
Llama-2-7B — Phase 12 (dmabuf) +1 ms
Llama-2-13B — Phase 10 (host staging) +26 ms
Llama-2-13B — Phase 12 (dmabuf) −4 ms

Negative overhead means multi-GPU is faster than single-GPU—doubled memory bandwidth outweighs transfer cost.

Design

  • Vendor-neutral — all changes use Vulkan 1.2+ spec features only. No driver-specific extensions except Phase 12 dmabuf (behind capability probe with clean fallback)
  • NVIDIA-safe — no changes to CUDA/NCCL paths. NVIDIA's Vulkan driver supports all APIs used
  • Host-staging model — cross-device transfers go through host memory (the only portable Vulkan approach). Phase 12 adds optional dmabuf for RADV
  • Single file — all changes in ggml/src/ggml-vulkan.cpp plus minimal integration in src/llama.cpp
