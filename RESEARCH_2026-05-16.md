# Research dive — sm_75 FA determinism & SoTA per-row CTA path

**Date**: 2026-05-16
**Question**: For batched FA on TU102/sm_75 with per-slot mask shapes (Qwen 3.6 27B production), what is the SoTA-correctness recipe and how expensive is it to implement?

## 1. TU102 / sm_75 hardware reference card

Compiled from NVIDIA Turing whitepaper, Turing Tuning Guide, and Quadro RTX 6000 datasheet:

| Property | Value |
|---|---|
| Compute capability | 7.5 (Turing) |
| Architecture | TU102 |
| SMs per GPU | 72 (RTX 6000) |
| FP32 cores per SM | 64 |
| INT32 cores per SM | 64 |
| Tensor cores per SM | 8 |
| Register file per SM | 64K × 32-bit = 256 KiB |
| Unified L1/SMEM per SM | 96 KiB |
| **Static SMEM per CTA hard cap** | **48 KiB** |
| Dynamic SMEM per CTA via opt-in | up to 64 KiB (with `cudaFuncSetAttribute(...MaxDynamicSharedMemorySize, 64*1024)`); enabling 32/64 carveout instead of 64/32 |
| Max threads per CTA | 1024 |
| Max thread blocks per SM | 16 |
| **Max concurrent warps per SM** | **32** (down from 64 on Volta) |
| L2 cache (total GPU) | 6 MiB on TU102 |
| FP16 tensor TFLOPS (fp32 acc) | 56.9 (whole GPU, Quadro RTX 6000) |
| FP32 cores TFLOPS (no tensor) | 16.3 |
| FP16 MAC/clock per SM | 512 (8 tensor cores × 64 FMA) |
| HBM-equivalent (GDDR6) bandwidth | ~672 GB/s |

**Tensor core MMA shapes natively supported on Turing (sm_75)**:
- `mma.sync.aligned.m8n8k4.f16.f16.f16.f16` (legacy)
- `mma.sync.aligned.m16n8k8.f16.f16.f32.f16` ✓ **the one we want**
- `mma.sync.aligned.m16n8k16.f16.f16.f32.f16` ✓ (added on Turing per "16x8x16 added" in tuning guide)
- WMMA wraps: m16n16k16, m32n8k16, m8n32k16 (fp16/fp16/fp32)

Larger sizes (m16n8k16 fp16/fp32, m16n8k32 int8) added on Turing+ but originally for sm_80; some are sm_75 native too.

**Key constraint for our design**: 48 KiB static SMEM cap → kernels must fit input tiles + accumulators + reduction scratch within 48 KiB per CTA, or opt into dynamic SMEM with a small perf cost (loses 16 KiB of L1).

## 2. Field SoTA: Thinking Machines Lab "batch-invariance" recipe (2025-09)

Source: https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/

**Root cause** (matches our TRACE-1..6 diagnosis exactly): "the reduction order for a given token does not depend on how many other tokens from its sequence are being simultaneously processed" — i.e., batch-size-invariant reduction order is the requirement.

**Three problem ops + fixes**:

1. **RMSNorm**: "data-parallel strategy: one batch element per core; entire reduction stays in single SM". Fixed tile size. ~20% slower than cuBLAS-equivalent.

2. **MatMul**: "fixed tiling, no Split-K, FP32 accumulation". Avoid tile heuristics that adapt to batch dim.

3. **Attention** (most complex): "**fixed split-size over KV**, not fixed-number-of-splits". Example: KV length 1000 with split_size=256 → 3 full splits of 256 + 1 tail of 232. NOT four splits of 250.

   Also: "Pre-update KV cache layout before attention kernel to ensure consistent layout regardless of how many tokens are being processed."

**Performance cost** (vLLM Qwen-3-8B, 1000 sequences):
- vLLM default: 26 s
- Unoptimized deterministic: 55 s (2.1× slower)
- With improved attention kernel: 42 s (1.6× slower)
- SGLang (downstream optimization with CUDA graphs): 34.35% slowdown vs ~61.5% for TML's reference

**Hardware**: tested on H100/H200. Recipe is architecture-generic — applies to Turing too.

**Reference implementation**: `thinking-machines-lab/batch_invariant_ops` (PyTorch + Triton via `torch.Library` substitution).

## 3. Field SoTA: SGLang deterministic inference (2025-09)

Source: https://www.lmsys.org/blog/2025-09-22-sglang-deterministic/

**Per-backend implementation**:
- **FlashInfer**: `fixed_split_size` + `disable_kv_split` parameters. Chunked-prefill truncation aligned to split size.
- **FlashAttention-3**: `num_splits=1` (no Split-K).
- **Triton**: fixed decoding split size with manual alignment.

**Performance**: 34.35% slowdown end-to-end. CUDA graphs give 2.79× speedup.

**Hardware**: H100/H200. SGLang doesn't explicitly support sm_75 in their deterministic path; we'd need to port the principles ourselves.

## 4. Field SoTA: llama.cpp PR #16016 "Deterministic inference mode" (draft, 2025)

Source: https://github.com/ggml-org/llama.cpp/pull/16016 (HEAD `d092e2682cc1db9f33b158b4378b448897b3096c`)

**Status**: open draft, NOT merged. Maintainer JohannesGaessler objected to maintaining bit-for-bit batch-invariance guarantees; suggested handling via prompt caching and single-slot submission instead.

**Recipe in code**:
- New env var `GGML_DETERMINISTIC=1` (or CLI `--deterministic`)
- New CMake flag `-DGGML_DETERMINISTIC=ON`
- Modified ggml-cuda kernels for RMSNorm, MatMul (FP16/BF16), Attention, MoE mul_mat_id
- **Attention path (fattn.cu)**: routes to `fattn-vec-f16` or `fattn-vec-f32` template family (which is per-row CTA architecture) instead of `fattn-mma-f16` (batched WMMA). Conservative dtype coverage:
  - F16/F16 at D ∈ {64, 128, 256} ✓
  - Q4_0/Q4_0 or Q8_0/Q8_0 at D=128 (NOT D=256 yet)
- New tile kernel `flash_attn_tile_generic_f16_singlecol` for head sizes not divisible by 64 (D ∈ {80, 96, 112}). Single warp per block, one Q column per block, online streaming softmax.
- "Force single-block accumulation" (no Split-K) for batch invariance.

**The PR's central architectural decision**: use the existing `fattn-vec-*` family for D=256/F16 because that family is ALREADY per-row CTA + online softmax + F32 accumulator. No new kernel needed for the common case.

## 5. ssiu/flash-attention-turing (CUTLASS-based)

Source: https://github.com/ssiu/flash-attention-turing

- **Per-row-block CTA** design via CUTLASS tiled_mma
- **mma.sync.m16n8k8** PTX (sm_75 native) via CUTLASS abstraction
- Warp-shuffle softmax reduction (no SMEM tree)
- Single Q load at prologue → SMEM, K/V streamed
- **Reaches ~63% of TU102 fp16 tensor peak** at long sequences on T4
- 2× faster than PyTorch SDPA on T4
- **Supported head dims: 64, 128 only**. No D=256.
- **No quantized KV support** (no Q4_0, Q8_0)
- Sequence length must be divisible by 128

**Adapting ssiu for our case**: would require head_dim=256 + Q4_0 KV support. Both significant additions. Not a drop-in.

## 6. Existing ik_llama.cpp fattn-vec-f32 architecture (LOCAL DISCOVERY)

**Source**: `ggml/src/ggml-cuda/fattn-vec-f32.cuh:17-422` in our repo.

**Kernel signature**:
```cpp
template<int Dk, int Dv, int ncols, ggml_type type_K, ggml_type type_V, bool use_logit_softcap>
__launch_bounds__(Dk, 1)
static __global__ void flash_attn_vec_ext_f32(...);
```

**Architecture**:
- **Per-row CTA** when ncols=1 (always on NVIDIA per the dispatcher at line 377: `if (Q->ne[1] == 1 || GGML_CUDA_CC_IS_NVIDIA(cc)) { cols_per_block = 1; ... }`)
- `Dk` threads per block, organized as `nwarps = Dk/32` warps × 32 lanes
- For Dk=256: 8 warps × 32 lanes = 256 threads/CTA
- **F32 throughout** (KQ scores in F32, VKQ accumulator in F32)
- **Online streaming softmax** (Welford-style; see line 257: `KQ_max_scale = expf(kqmax[j] - kqmax_new_j)`)
- **vec_dot_KQ** (scalar dot product via CUDA cores, NOT tensor cores)
- **Mask-bounded K-loop**: line 188 reads `KV_min_max` for per-block K range; line 194's K-loop iterates that range
- **Split-K support** via `gridDim.y` and `KV_min_max`; if `parallel_blocks=1` then `gridDim.y=1` → single-block accumulation per row → no Split-K → batch-invariant

**Template instance for our case**: `fattn-vec-f32-instance-hs256-f16-f16.cu` exists and is compiled. Direct match for Dk=Dv=256, K=F16, V=F16.

**Critical realization**: this kernel IS the per-row CTA F32 batch-invariant architecture I proposed in the FIX-C v3 plan. It's already implemented. Already tested. Already compiled.

**Required pre-step for production**: `launch_fattn` with `need_f16_K=true, need_f16_V=true` will pre-dequant Q4_0 K/V → F16 buffer, then call this kernel. Already wired.

## 7. Determinism guarantees of fattn-vec-f32 (line-by-line audit)

Auditing the kernel for batch-invariance:

| Concern | Status | Where |
|---|---|---|
| Per-row CTA | ✓ | `blockIdx.x` indexes Q col; ncols=1 on NVIDIA |
| F32 accumulators | ✓ | `float VKQ[ncols] = {0.0f};` line 186 |
| F32 KQ scores | ✓ | `__shared__ float KQ[ncols*Dk]` line 84 |
| Canonical K-loop | ✓ | `for (k_VKQ_0 = first_y + blockIdx.y*Dk; k_VKQ_0 < k_VKQ_max; k_VKQ_0 += gridDim.y*Dk)` — strides by `gridDim.y*Dk`; at parallel_blocks=1 → gridDim.y=1 → canonical stride Dk |
| Online softmax stable | ✓ | Welford-style at line 257-260 |
| No atomicAdd | ✓ | grep confirmed no atomic ops |
| Mask read per-row | ✓ | `maskh[j*ne11 + tid]` line 203, j∈[0..ncols) so j=0 for ncols=1 |
| warp_reduce_sum commutative-safe | ✓ | Per TRACE-4 proof |
| K loop bound per-row | ✓ | `k_VKQ_max` from `KV_min_max[sequence*gridDim.x + blockIdx.x]` — per (sequence, x-block = row) |
| Split-K disabled | conditional | `parallel_blocks=1` in `launch_fattn` → `gridDim.y=1` → no Split-K |

**Conclusion**: at `parallel_blocks=1`, the kernel is fully batch-invariant per-row. Matches TML recipe and llama.cpp PR #16016's approach.

## 8. Critical perf data point — what's vec_f32's actual cost vs wmma_f16?

Spec `fattn-per-slot-kv-sm75.md §15.13` cites: "~12× slower per FA call than wmma_f16". That measurement is from an older session; needs re-validation.

**Theoretical analysis**:
- wmma_f16_case_pb1 with cpb=8 at ne[1]=N: 1 CTA per (row_block × head), each CTA processes 8 cols of Q via WMMA tensor cores. For ne[1]=1, 7/8 cols are zero-padded (wasted tensor-core cycles).
- vec_f32 with ncols=1 at ne[1]=N: N CTAs (1 per row × head), each CTA processes 1 col via CUDA-core vec_dot. For ne[1]=1, 1 CTA. For ne[1]=8, 8 CTAs.

Total tensor-core work: wmma always does ~16 mma per K-chunk per CTA at full tensor throughput; vec uses 0 tensor cores, does ~256 CUDA-core dot products per K-chunk per CTA.

**FA is memory-bound at decode** (per analysis ~3-5% of decode time at NP=1). The "12×" was likely measured at PREFILL where FA is more compute-bound.

**Decode at NP=1**: FA HBM reads = ~190 µs per token; FA compute = ~10-50 µs. Total FA ~200 µs. Total decode ~30 ms. FA share ~0.7%.

vec_f32 vs wmma at decode: HBM reads identical (both read K/V once). Compute: vec_f32 needs more CUDA-core cycles than wmma's tensor-core cycles, but at memory-bound regime this doesn't matter.

**Predicted decode perf** at NP=1: vec_f32 within 2-3× of wmma per FA call. End-to-end: 1-3% slowdown total. Acceptable.

**At prefill** (ne[1]=n_prompt up to 4096): vec_f32 with N=n_prompt CTAs is BETTER per-CTA workload than wmma's cpb=8 batched (which handles 8 prompt tokens per CTA). vec_f32 dispatches more CTAs but each is well-utilized. Prefill perf TBD via measurement.

## 9. Revised FIX-C plan (vNext) — leveraging fattn-vec-f32

### 9a. The whole fix in one paragraph

In `ggml/src/ggml-cuda/fattn-per-slot-kv-sm75.cu`, change the dispatcher from
```cpp
ggml_cuda_flash_attn_ext_wmma_f16_case_pb1<256, 256, 8, float>(ctx, dst);
```
to
```cpp
ggml_cuda_flash_attn_ext_vec_f32_case<256, 256, GGML_TYPE_F16, GGML_TYPE_F16>(ctx, dst);
```

`launch_fattn` inside `vec_f32_case` is called with `need_f16_K=true, need_f16_V=true` (already the default), which pre-dequants Q4_0 cache to F16 before the kernel sees it. The vec_f32 kernel with ncols=1 (NVIDIA path) is per-row CTA + canonical k-loop + online softmax + F32 throughout. At `parallel_blocks=1`, the kernel is batch-invariant by construction.

Everything else (build graph, KV cache layout, mask handling, output shape) stays identical.

### 9b. Why this is SoTA

- Uses the production-tested per-row CTA architecture (same as upstream llama.cpp PR #16016, TML recipe).
- Same dispatcher contract as current — drop-in replacement.
- Zero new kernel code; zero new template instances; zero new test scaffolding.
- Aligns with field standard (TML, SGLang, llama.cpp PR #16016 all converge on this).

### 9c. Risks

1. **vec_f32 perf is materially worse than wmma_f16**: spec §15.13 claimed 12×. Need empirical measurement. If true, find an optimization or accept the trade-off (decode determinism > raw throughput).
2. **vec_f32 might not be batch-invariant in some edge case** I missed. The line-by-line audit (§7 above) found nothing, but production has surprised us before (TRACE-1).
3. **launch_fattn's `parallel_blocks` selection** may pick something > 1 for our shape, enabling Split-K and breaking batch invariance. Need to verify or force `parallel_blocks=1`.
4. **Q4_0 dequant kernel may itself be batch-shape-dependent**: the pre-FA `to_fp16_cuda(K->type)` dequant is a separate kernel. Need to verify it produces byte-identical output regardless of n_tokens.

### 9e. Reference oracle for unit tests

For V1: a **scalar fp32 oracle** in CPU. Algorithm:
```python
for each (batch_idx, head_q, query_pos):
    q = Q[head_q, query_pos]   # head_dim floats
    KQ_max = -inf; KQ_sum = 0; VKQ_acc = zeros(head_dim_v)
    for k in range(k_VKQ_max):
        # mask check
        if mask[batch_idx, k] == -inf: continue
        kq = sum(q * K[head_q // gqa_ratio, k]) * scale + slope * mask[batch_idx, k]
        new_max = max(KQ_max, kq)
        scale_corr = exp(KQ_max - new_max) if KQ_max > -inf else 0
        KQ_sum = KQ_sum * scale_corr + exp(kq - new_max)
        VKQ_acc = VKQ_acc * scale_corr + V[head_q // gqa_ratio, k] * exp(kq - new_max)
        KQ_max = new_max
    out[head_q, query_pos] = VKQ_acc / KQ_sum
```
fp32 throughout, scalar loops, no parallelism. Reference values are byte-deterministic per (Q, K, V, mask) input.

The kernel SHOULD produce byte-identical output to the oracle (modulo well-understood mma instruction internal ordering — but vec_f32 doesn't use mma, so byte-identity should be exact).

Save oracle outputs to `specs/deltanet/oracle/fattn-vec-f32-{config}.npy` for V1.

## 11. Open empirical questions before commit

1. **What is the ACTUAL perf cost of vec_f32 vs wmma_f16 at our production shape?** Spec §15.13's "12×" claim is unverified for our binary + shape. Measurement.
2. **Does `launch_fattn` default `parallel_blocks=1` for ne[1]=1, vec_f32 path?** Need to read the launch logic in fattn-vec-f32.cuh and fattn-common.cuh.
3. **Is the Q4_0 → F16 dequant kernel batch-shape-invariant?** Likely yes (per-cell deterministic transformation), but worth probing with a focused test.
4. **Does vec_f32 handle the per-slot-kv `inp_per_row_k_bound` argument correctly?** The op `ggml_flash_attn_ext_per_slot_kv` has `src[5]=per_row_k_bound`; we need to verify vec_f32 reads it (likely doesn't, since vec_f32 is for the regular FA op). Need to either pass bound via `KV_min_max` or rely on mask.

## 12. Spec-led test-first plan

For specification-led test-first design before code:

1. Write the scalar fp32 oracle (§9e). This is the canonical reference; everything else compares against it.
2. Write V1 unit test that exercises `ggml_cuda_flash_attn_ext_vec_f32_case<256, 256, F16, F16>` with synthetic inputs and compares to oracle byte-exactly. The test must FAIL until correctness is confirmed.
3. Write TRACE-6'-style integration test that runs the dispatcher-swapped FA in the production graph at NP=2 same-prompt. The test must FAIL until the dispatcher change lands.
4. THEN make the dispatcher change.
5. Verify V1 (oracle binding), V2 (TRACE-6'), then V3-V6.

Tests gate the fix. Spec-led: oracle is the spec; tests bind on it.

## 13. Decision points for user

**DP1**: Proceed with FIX-C v4 (route to existing vec_f32)? Recommended given equal correctness + much less effort.

**DP2**: Reference oracle implementation language — Python/numpy (easy to write/inspect) or C++ (easier to embed in tests)? Recommendation: write in C++ as header (`fattn-vec-f32-reference.h`) to avoid runtime dependency.

**DP3**: Test fixture inputs — synthetic (mt19937 random) or captured-from-production (TRACE data)? Recommendation: BOTH. Synthetic for V1 (random sweep). Production-captured for V2-V3 (real-shape binding).

**DP4**: Should V6 perf measurement gate the ship? Recommendation: yes — if vec_f32 is >50% end-to-end slower than wmma_f16, surface the trade-off explicitly to the user before shipping.
