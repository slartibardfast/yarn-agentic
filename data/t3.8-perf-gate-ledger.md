| 1 | 27.62 | 24.45 | 25.15 | 65.437 | 18.539 | 77.9% | 11 graphs / 308 nodes / 79.5 KB host (dev0); 16 / 350 / 90.3 (dev1) |
| 2 | 27.70 | 24.47 | 25.19 | 65.374 | 18.481 | 77.9% | same |
| 3 | 27.86 | 24.44 | 25.19 | 65.456 | 18.375 | 78.1% | same |

**Mean S_TG = 27.73 t/s aggregate, σ = 0.12, CV = 0.43%.**

**Critical finding:** every M2 run emits `llama_decode_internal: interleaved batch — sub-batching by seq_id (first run=1 of 512 tokens, count=1)` at the first TG tick. This means `llama-batched-bench`'s token-major batch layout (`common_batch_add(batch, 0, pp+i, {j}, true)` at `examples/batched-bench/batched-bench.cpp:175-186` — token-outer, seq-inner) is detected as `QNEXT_SEQ_INTERLEAVED` and **falls back to single-seq sub-batching at `src/llama.cpp:5789`**. This means:

1. T3.5's multi-seq dispatch path is **NOT exercised** by `llama-batched-bench`.
2. The +15% (vs ledger row 21's 23.96) / 0% (vs ledger row 6's 27.73) uplift is from secondary T3 effects (K/V WRITE set_rows, layout cleanup) on the sub-batched fallback path, not the load-bearing 4D dispatch.
3. **M2 cannot bind the GP3.i gate.** That requires M3 (server-driven, where `process_batch_tokens` converts interleaved→contiguous before dispatch).

**Stall %:** 77.9% — the bench is overwhelmingly prefill-bound at this `-npp 200 -ntg 64` shape with 8 slots. This shape is a 200-token cold prefill per slot followed by only 64 decode tokens; real workloads have much higher decode share.

The unified multi-seq dispatch at 93% firing rate **does not deliver measurable additional throughput at decode shape** vs sub-batched single-seq fallback. This generalises the PHASE doc Tier 2 finding (CUDA graphs ≈ 0% at our shape) to dispatch packing: at decode-only n_tokens=1-per-seq, batching the seqs into one tick vs running them in 8 sub-ticks delivers no measurable uplift on PSKV-singlewarp + Q4_0 KV + sm_75.

## T3.8.E — Evaluation

**Gate verdict (binding numbers):**

| Gate | Target | Measured | Result |
|---|---|---|---|
| GP3.i conservative (vLLM-comparable) | ≥ 100 t/s | M2 mean 27.73 t/s | **FAIL (28% of gate)** |
| GP3.i stretch (vLLM-comparable) | ≥ 130 t/s | M2 mean 27.73 t/s | **FAIL (21% of gate)** |
| Production-realistic Hadamard | ≥ 90 t/s | M3 mean 26.49 t/s | **FAIL (29% of gate)** |

**Honest framing: the GP3.i gate was anchored on a wrong theory of which lever produces uplift.** The PHASE doc reasoned that **unified-stream multi-seq dispatch** would close the gap to vLLM's 154.77 t/s. T3.5 delivers that dispatch at 93% firing rate (M3 dispatch counter). The dispatch itself delivers ~0% throughput uplift at decode shape on PSKV-singlewarp + Q4_0 KV + sm_75.

**Why the theory missed:**

- **Empirical:** M2 (fallback path, T3.5 bypassed) ≈ M3 (T3.5 active at 93%) within Hadamard tax. The sub-batched single-seq fallback and the unified multi-seq dispatch deliver the same throughput at our shape. This generalises the PHASE doc Tier 2 finding (CUDA graph reuse ≈ 0% at our shape) to dispatch packing.
- **Mechanistic:** at decode n_tokens=1-per-seq, the kernel work is grid-parallel per-seq either way; the launch + dispatch overhead is amortised below the per-kernel runtime; memory traffic dominates and isn't reduced by packing. The "Tier 3 unified-stream dispatch" lever in the PHASE doc estimated +3.6×–4.7× from "kernel batching, NOT graph reuse" — that assumption did not hold for our kernel + shape.
- **vLLM's lever is different:** per `project_continuous_batching_vs_perslot_dispatch` and PHASE doc OpenQ-C, vLLM's 154.77 t/s comes from **continuous batching / chunked-prefill admission** — splicing new-request prefill into running-decode ubatches. That's PHASE doc Tier 4, never lifted. T3 closure does not address it.

**Uplift attribution:**

- (M2 - ledger row 21): 27.73 - 23.96 = +15.7% if anchored to 2026-05-21.
- (M2 - ledger row 6): 27.73 - 27.73 = +0.0% if anchored to 2026-05-20.
- The two prior "baselines" disagree by 14% on the same config; with locked clocks (M2 CV 0.43%) we cannot adjudicate which was the noisy run. **T3 kernel-path throughput change is within historical variance; not a measurable uplift on TG aggregate.**
- (M3 - M2): −4.5% Hadamard tax, as expected.
- (M3 - hypothetical "T3.5-off") would require a counter-disabling rebuild. Not run this session; the M2 fallback-path number is the closest available proxy (Hadamard-adjusted, ~27.8 t/s) — i.e. T3.5 enabled the multi-seq dispatch path at the same throughput, no regression.

**Variance characterization:**

| Config | N | Mean t/s | σ | CV |
|---|---|---|---|---|
| M1 NP=1 | 3 | 24.04 | 0.026 | 0.11% |
| M2 NP=8 no-Hadamard | 3 | 27.73 | 0.12 | 0.43% |
| M3 NP=8 Hadamard | 3 | 26.49 | 0.037 | 0.14% |

All three configs are well-disciplined under locked GPU clocks. CV < 0.5% means single-shot bench numbers ARE reliable under this regime; the 14% day-to-day historical variance in ledger rows 6 vs 21 was almost certainly unlocked-clock or commit-shift confound. **Future T3.x perf work should require locked clocks for binding measurements.**

**Stall % analysis (Tier 4 decision):**

- M2: T_PP / (T_PP + T_TG) = 65.44 / 83.98 = **77.9%** prefill-bound at this shape (`-npp 200 -ntg 64`).
- M3: each of 8 slots does its own 200-token prefill via /v1/completions; concurrent prefills cause server-side queueing. Hard to extract a single "stall %" cleanly without per-slot tick tracing.
- The PHASE doc's Tier 4 trigger ("if conservative misses AND stall > 30%, justify Tier 4") fires: M2 stall is 77.9% AND conservative misses by 3.6×. **Tier 4 (chunked-prefill admission / continuous batching) is the justified next perf lever**, not more dispatch-layer engineering.

**Graph-pool VRAM probe data (informs future I.b.2 revisit):**

- M1 NP=1: 7–10 graphs / 166–213 nodes / 42–55 KB host per device.
- M2 NP=8: 11–16 graphs / 308–350 nodes / 79–90 KB host per device.
- M3 NP=8 server: 11–22 graphs / 326–400 nodes / 84–103 KB host per device.

Pool grows with parallelism but remains bounded (< 25 graphs, < 400 nodes, < 110 KB host per device — well within "no runaway" territory). I.b.2 bailout-drop revisit doesn't have a VRAM-pressure justification.

## T3.8.C — Closure verdict

**GP3.i gate result: FAIL on all three sub-gates.**

T3 Framing B delivered correctness completeness: multi-seq dispatch infrastructure, K-shift / defrag under graph-split, generic CUDA Q→Q cpy, audit-grade specs + tests + closure for T3.6, and the VRAM probe instrumentation. **But the throughput uplift the PHASE doc projected from these correctness changes did not materialise at our model + kernel + shape.** Honest result: the gate framing was based on a wrong theory of where the throughput is locked away.

**Recommended next action: Tier 4 (continuous batching / chunked-prefill admission).** PHASE doc OpenQ-C names this as the contingency; M2's 77.9% prefill stall and the M2≈M3 equivalence at decode shape jointly justify it. The work would splice new-request prefill into running-decode ubatches, eliminating prefill stalls — closer to vLLM's continuous-batching scheduler. Token budget for Tier 4 is a fresh phase, out of scope here.

**Do NOT revert T3 correctness changes** — they are correctness infrastructure, not "perf attempts." Multi-stream KV is necessary for future ctx + parallelism scaling regardless of which throughput lever is pursued next; the audit-grade test + spec coverage gates regressions; the VRAM probe is permanent measurement plumbing.

**Branch left clean. Production profile (`profiles/active.sh`) unchanged this session** — the current production NP=2 + DFlash + Hadamard configuration is unaffected by T3.8 outcome.
