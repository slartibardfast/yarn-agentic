# Slot-0 trace: NP=8 intra-NP determinism (2026-05-16)

## Empirical picture

Production server (no cb_eval), 13-token prompt, 32 decode steps, NP=8 with 8 concurrent same-prompt requests. Env: singlewarp + shape_invariant_dispatch + cublas_pin.

### Without `STRICT_SEQUENTIAL`

Three sequential runs:
- Run 1: only slot 7 matches NP=1 baseline. Slots 0-6 diverge.
- Run 2: only slot 4 matches. Slots 0-3, 5-7 diverge.
- Run 3: only slot 4 matches. Slots 0-3, 5-7 diverge.

Run-to-run variability — which slot is "OK" shifts between runs. Concurrency-induced non-determinism layered on top of an underlying slot-index issue.

### With `LLAMA_FATTN_STRICT_SEQUENTIAL_DECODE=1`

Two runs (one slot per batch enforced):
- Run 1: slot 0 OK, slots 1-7 FAIL.
- Run 2: slot 0 OK, slot 4 OK, slots 1-3, 5-7 FAIL.

**Slot 0 is now CONSISTENTLY OK** across runs. Slot 4 sometimes OK. Slots 1, 2, 3, 5, 6, 7 always FAIL.

### Bisecting NP boundary

- NP={1, 2, 3, 4}: all slots byte-identical to NP=1 (fully deterministic) ✓
- NP=5: slot 3 FAIL, others OK ← boundary
- NP=6: slot 0 diverges from NP=1 (cross-NP slot 0 byte-identity LOST)
- NP=7, NP=8: more slots diverge

The boundary is around NP=5. Above that, intra-NP determinism breaks.

## Reframe: it's NOT a "first slot is special" effect

Earlier I posited slot 0 was special. With STRICT_SEQUENTIAL we see slot 0 IS consistently OK, so it looks special. But:
- Without STRICT_SEQUENTIAL, slot 0 is NOT consistently OK across runs
- The actual pattern is "request-to-slot assignment varies + slot-index addressing bug for index > 0 + concurrency layer"

STRICT_SEQUENTIAL just removes the concurrency layer, exposing the underlying deterministic bug.

## What this rules out

- **Atomic-op non-determinism**: ruled out. With STRICT_SEQUENTIAL the result becomes deterministic per run.
- **Race conditions on shared resources**: still possible but smaller; STRICT_SEQUENTIAL handles much of it.
- **GPU scheduling**: ruled out as a sole cause; deterministic addressing bug still exists.

## What remains

A **slot-index addressing bug for slot_id > 0** somewhere in the production graph. Suspects:

1. **DeltaNet recurrent state buffer offset** at `state_seq_id_local * state_row_size` — the addressing arithmetic LOOKS correct (line 332 in llama-delta-net.cpp). But the actual memory layout might have alignment or stride issues at slot > 0.

2. **KV cache region for slot_id > 0**: prefill writes K/V at offsets per slot. If offsets are computed slightly wrong for slot > 0, the cache contents differ from what FA expects.

3. **Hadamard rotation kernel**: applied per K/V before cache write. The kernel doesn't take a slot index but the input tensor's view is per-slot.

4. **`ssm_conv_f32_kernel_nc4` fallback** — has a `t == 0` special case (line 557 in ssm-conv.cu) that could trigger differently for slot 0 vs slot 1+. Per the dispatch path, this fallback should early-return when fast paths handle multi-seq; but if it ever runs for some edge config, it would corrupt slots 1+.

## Most productive next probe (CY.F.9)

Bypass the concurrency layer. Run NP=1 server with sequential same-prompt requests — confirm slot 0 always produces canonical output. Then run NP=2 server with sequential requests (each goes to slot 0 alone). Compare. If NP=2 slot-0-alone matches NP=1, the bug is concurrent slot scheduling.

If NP=2 sequential ALSO produces canonical, the slot-1-when-active issue requires hitting slot 1 specifically. Need a way to force a request to a particular slot ID — server doesn't expose this in the API.

Alternative: **write a focused C++ test that calls llama_decode with seq_id=1 directly** (not via HTTP). Initialize a context with n_seq_max=8, prefill seq_id=0 ALONE, get its output. Then prefill seq_id=1 ALONE (after slot 0 finishes), get ITS output. If slot 1's output differs from slot 0's despite identical input, the bug is in the per-slot graph build OR per-slot-id kernel addressing.

## What we know works

- NP={1, 2, 3, 4} is fully byte-deterministic. **Production-ready for these slot counts.**
- Cross-NP slot 0 byte-identity at NP=1 vs NP=2 vs NP=4: confirmed.
- Singlewarp FA per-slot kernel: deterministic per CTA.

For production rollout, recommendation: **deploy at NP≤4** until the slot-id > 0 addressing bug is found and fixed.
