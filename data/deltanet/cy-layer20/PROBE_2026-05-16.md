# Phase CY.F probe: NP=4 vs NP=8 layer-20 boundary (2026-05-16)

## Setup

Production env stack (`singlewarp` + `shape_invariant_dispatch` + `cublas_pin`).
Same 13-token prompt across all slots. Single decode step at NP=4 / NP=8.
Capture mechanism: raw `cparams.cb_eval` matched-name (cy-layer0-capture test).

## Key conclusion

The DeltaNet **recurrent state** at layer 20 is **byte-identical** between NP=4 and NP=8 for slot 0 (`state_in-20` ndiff=0). Phase D buffer-addressing hypothesis (different n_seq_max → different state buffer layout) is **falsified**.

The divergence enters at `qkv_mixed-20000` — the first projection inside the layer-20 DeltaNet (Wqkv @ residual). Since the weight Wqkv is identical and (per d1-capture) the residual `l_out-19` is byte-identical, the Wqkv matmul should produce byte-identical output.

## Open contradiction (capture-mechanism discrepancy)

- **d1-capture** (`llama_set_dflash_extract_layers` + `dflash_extract_buf` with dtype branching): says `l_out-19` is byte-identical NP=4↔NP=8 slot 0. Layer 20 is the first diff.
- **cy-capture** (raw `cparams.cb_eval`): says `qkv_mixed-20000` differs by 1.08e-01 at layer 20 (implies the residual feeding Wqkv differs).

Both captures fire on the same logical tensor at the same `cb()` call site. They produce DIFFERENT values for the same tensor.

Hypothesis: the two mechanisms read at different graph-evaluation points relative to in-place op modifications. The d1 mechanism reads the residual at a point where the tensor has been overwritten by an in-place op aliasing the source. The cy mechanism reads at a point before the in-place op fires (or vice versa).

## What's eliminated by this probe

1. **DeltaNet recurrent state corruption at higher `n_seq_max`** — state_in-20 is byte-identical.
2. **`n_seq_max=8` buffer-addressing issue** — kernel correctly reads slot 0's state regardless of buffer size.
3. **State accumulation drift through prefill** — slot 0's state through 13-token prefill is the same at NP=4 and NP=8.

## What's NOT eliminated

1. **Wqkv matmul at ne[1]=4 vs ne[1]=8**: same mmq_x_best=8 by code, but empirically diverges per cy-capture. CY.F.1 test at K=27648/N=5120 with M ∈ {1,4,8} PASSED — but that test uses small synthetic weights, not production Q4_0_AR16 weights. May need test with actual production weights extracted from the model.
2. **`ggml_reduce` cross-device sum** at higher concurrency.
3. **Capture-mechanism artifact**: maybe cy-capture's qkv_mixed-20000 is unreliable in the same way its l_out captures are.

## Next subprobe (CY.F.5)

Direct unit test: extract layer-20 wqkv weight from the production GGUF and a slot-0 residual. Drive `ggml_mul_mat(wqkv, residual_in)` at ne[1]=4 and ne[1]=8 with the same slot-0 column. Compare slot-0 output. If FAIL: this is the source. If PASS: cy-capture is unreliable for ALL intermediates, not just l_out.
